# Feature 0010: Capture user outcomes and feed them back into model/risk-score refinement

## Context

This feature implements the product-brief functional requirement, verbatim: **“Capture user outcomes and feed them back into model/risk-score refinement.”**  
Feature **0009** already records operator **issue workflow state** (`fixed`, `monitoring`, `deferred`, `ignored`) and append-only **`issue_state_events`**, but it does **not** define a structured **outcome signal** for downstream **model training/calibration** or **risk-score tuning**, nor does it connect those signals to **`detections`** / **`maintenance_recommendations`** / **`risk_rules`** evaluation paths.

This plan adds **first-class outcome capture** (immutable or append-only rows, queryable by MLOps and internal analytics) and a **bounded, explicit “feedback channel”** into the existing **rule-based recommendation score** path. It does **not** implement in-repo **SageMaker training jobs**, **XGBoost service**, or **automatic rewriting of `risk_rules` rows** from learned models; those remain external consumers of exported/queryable data unless a later feature narrows that scope.

## Files and modules to add or change

**Domain and persistence (new)**

- `backend/app/models/outcome_feedback.py` (new; name may be `operator_outcome` or `user_outcome`—pick one term and use consistently in DB table + Python)
  - Append-only row: stable `id`, `created_at`, optional `actor` / `updated_by` string (align naming with `IssueState`).
  - **Scope:** `org_id` (nullable UUID, same semantics as issues), `asset_zone_id`, `issue_key` (string; reuse `build_issue_key` normalization where the client supplies type/class/subtype instead of raw key).
  - **Anchors (nullable FKs where allowed):** `target_inspection_id` (FK `inspections.id`), optional `issue_state_id` / `issue_state_event_id` (FK into `issue_states` / `issue_state_events`) to tie feedback to a specific transition or row without replacing 0009 behavior.
  - **Model-relevant refs:** optional JSON `detection_refs` (list of `{ "detection_id": uuid }` or similar) and/or single `primary_detection_id`—must be consistent with how workers persist **`Detection`** rows (`backend/app/models/detection.py`: `model_name`, `model_version`, `class_name`, `confidence`, frame link via `frame_id`).
  - **Recommendation context (risk path):** optional `maintenance_recommendation_id` is awkward because recommendations are **replaced** each worker run (`delete` in `run_recommendations_for_inspection` in `backend/app/services/recommendation_engine.py`); prefer storing **denormalized snapshot** fields at capture time: `captured_priority_label`, `captured_priority_score`, `target_inspection_id`, `asset_zone_id`—so feedback survives recommendation regeneration. Document this explicitly in API docs.
  - **Typed outcome:** constrained string or enum for v1 categories that map cleanly to both use cases, e.g. split into `outcome_kind` = `model_label` \| `risk_priority` \| `general` and `outcome_code` (small fixed set per kind: e.g. `false_positive`, `false_negative`, `severity_understated`, `severity_overstated`, `priority_too_high`, `priority_too_low`, `confirmed`, `other`) plus optional `detail` / `notes` text and optional `context` JSON (correlation id, UI version).
  - Indexes for export filters: `(created_at)`, `(org_id, created_at)`, `(target_inspection_id)`, `(asset_zone_id, issue_key)`, `(primary_detection_id)` if present.

## Algorithms and behavior (step-by-step)

**A. Capture (submit)**

1. Client sends `outcome_kind`, `outcome_code`, `asset_zone_id`, and identity for `issue_key` (direct key or `detection_type` + `class_name` + optional `subtype` mirroring **`PUT /issues/state`** resolution in `backend/app/api/routes/issues.py` via `_resolve_issue_key` + `build_issue_key`).
2. Server validates enums; rejects unknown combinations with **422**.
3. Server validates optional FKs (`target_inspection_id`, detection ids, issue state ids) — **404** or **422** policy should match existing APIs (`Inspection` missing behavior in `issue_state_service` uses **404** for missing inspection on `last_target_inspection_id`; stay consistent).
4. If recommendation snapshot fields are omitted but `target_inspection_id` + zone are present, optional **best-effort** enrichment: single `SELECT` of the current **`MaintenanceRecommendation`** row for that inspection + zone (if any) to populate denormalized score/label at insert time—if none, store nulls. Document that this is a snapshot, not a live join.
5. `INSERT` only; no updates (append-only).

**B. Export / consumption for model refinement (offline path)**

1. MLOps jobs query **`GET`** with time window + optional `model_name`/`model_version` filters.
2. Join path for training labels: starting from `primary_detection_id` / `detection_refs`, load **`Detection`** rows to recover `frame_id`, bbox, `model_name` / `model_version`, and then resolve frame S3 keys via **`Frame`** (`backend/app/models/frame.py`) for crop extraction **outside** this service (SageMaker pipeline reads DB or an export dump). This plan only requires **stable foreign keys + model metadata** on the feedback row or reachable via joins.

**C. Feed back into risk-score refinement (online, bounded)**

1. During `run_recommendations_for_inspection` in `backend/app/services/recommendation_engine.py`, after zone grouping and before or inside each `score_zone` call, load **aggregates** for that `(org_scope or org_id policy, asset_zone_id, issue_key)` over a configured lookback (count by `outcome_code` where `outcome_kind` includes risk-related codes).
2. If counts are below `min_feedback_samples`, apply **no** adjustment.
3. Otherwise compute a **scalar adjustment** bounded by settings (e.g. subtract up to N points for sustained `priority_too_high` / false-alarm patterns; add small bump for sustained under-prioritization signals). Merge into rationale `factors` as a synthetic factor dict (`kind`: e.g. `operator_feedback`) so **`maintenance_recommendations.rationale`** remains explainable.
4. Ensure adjustment composes **after** base score and **before or after** `apply_risk_rule_effects` in `backend/app/services/risk_rule_eval.py` — pick one order and document it (recommend: apply feedback prior **after** persisted `risk_rules` so human rules remain authoritative, or **before** rules if feedback should be amplifiable by rules—product choice; state both options in implementation ticket).

## Phases

**Phase 1 — Data layer + APIs**  
New table(s), Alembic, schemas, service, `POST`/`GET` routes, tests, and documentation. No change to `score_zone` yet (or guard all scoring changes behind a default-off setting).

**Phase 2 — Scoring integration**  
Settings + aggregation queries + `recommendation_rules.py` / `recommendation_engine.py` wiring + tests proving bounded influence and rationale visibility.

Phases **2A/2B** (parallel UI vs backend) are **not** listed here because this repository slice is backend-only; a dashboard would be a separate client/repo task once the API is stable.

## Explicitly out of scope

- Training or hosting **XGBoost** / **PyTorch** jobs inside `backend/`.
- Auto-editing **`risk_rules`** JSON from learned policies.
- Replacing **`maintenance_recommendations`** replace-semantics with versioning (would be a separate schema feature).
- Cryptographic non-repudiation or full legal audit beyond append-only rows + actor string.
