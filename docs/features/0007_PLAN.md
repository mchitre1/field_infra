# Feature 0007: Produce prioritized maintenance recommendations with rationale and suggested SLA windows

## Context

This feature implements the product-brief functional requirement, verbatim: **“Produce prioritized maintenance recommendations with rationale and suggested SLA windows.”**  
The backend already persists **detections** (type, `class_name`, `confidence`, bbox, optional attributes), **alignment pairs** and **change events** (including `appeared` / `disappeared` / `persisted`), **progression metrics** (e.g. crack growth rate, vegetation deltas), and **temporal insight** read APIs (change map, timeline, trends). There is **no** decision layer yet that fuses these signals into ranked work items, human-readable **rationale**, or **suggested SLA** (target completion window) per asset zone or inspection.

The brief’s adjacent requirement **“Support configurable risk rules by asset type, severity, and environmental conditions”** is **not** fully specified here; v1 should use **settings-backed numeric thresholds** (and optional JSON rule hints) so the recommendation engine can later swap in RDS-backed rules or ML scores without redesigning the row shape.

## Files and modules to add or change

**Configuration**

- `backend/app/core/config.py`
  - Recommendation and SLA tuning, for example:
    - score weights for high-confidence defects, `appeared` change events, positive progression rates/deltas above configured floors,
    - band cutoffs mapping scalar **priority score** → label (`critical` \| `high` \| `medium` \| `low`),
    - **SLA policy**: days from **target inspection effective time** (`capture_timestamp` or `created_at`, same precedence as progression/temporal insights) to suggested `sla_target_at` per band (e.g. critical 7d, high 30d, medium 90d, low 180d),
    - optional max recommendations per inspection / per `asset_zone_id` for response size caps.

**Domain and persistence**

- `backend/app/models/maintenance_recommendation.py` (new)
  - One row per recommendation line item for audit/traceability, for example:
    - `id` (UUID),
    - `target_inspection_id` (FK `inspections.id`, indexed),
    - `asset_zone_id` (string, indexed),
    - `priority_rank` (int, 1 = highest within that inspection run),
    - `priority_label` (string, aligned to configured bands),
    - `priority_score` (float),
    - `title` / `action_summary` (short string for “what to fix”),
    - `rationale` (JSON: ordered list of structured factors, each with `kind`, `message`, and optional refs such as `detection_id`, `change_event_id`, `progression_metric_id`, `metric_name`, `value`),
    - `sla_target_at` (timestamptz, absolute suggested deadline),
    - `sla_days_suggested` (float or int, echo of policy input used),
    - `created_at`.
  - Optional uniqueness constraint or idempotent delete pattern: `(target_inspection_id, asset_zone_id, action_summary)` or “delete all for `target_inspection_id` then insert” to match progression’s replace semantics.

## Algorithms and behavior (step-by-step)

**A. Eligibility and scope**

1. Target inspection exists and has completed at least **detections** and **alignment** (recommendations may run after progression with score contributions zero when no progression rows).
2. Collect distinct `asset_zone_id` values from: `ChangeEvent` and `Alignment` for `target_inspection_id`, plus `Detection.asset_zone_hint` / derived zone if the engine already attributes detections to zones (reuse `asset_zone` derivation patterns from alignment pipeline where possible).

**B. Signal fusion (per zone)**

1. **Detections:** max confidence per `class_name` / `detection_type`, count of high-severity classes; add rationale lines when above configured floors.
2. **Change events:** `appeared` / `disappeared` / other `event_type` strings contribute additive score bumps and explicit rationale entries referencing `change_event_id`.
3. **Progression metrics:** compare `metric_name` and `value` to thresholds (e.g. crack growth rate above X) to add factors referencing `progression_metric_id`.

**C. Priority score and ordering**

1. Sum weighted contributions into `priority_score` (deterministic, documented weights in settings).
2. Sort zones (or per-zone issue rows if multiple actions per zone in v1) descending by `priority_score`, then stable secondary key (`asset_zone_id`).
3. Assign `priority_rank` 1..N and map score to `priority_label` using configured band edges.

**D. SLA window**

1. Let `t_eff = coalesce(inspection.capture_timestamp, inspection.created_at)` for the **target** inspection.
2. `sla_days_suggested = policy_days[priority_label]`.
3. `sla_target_at = t_eff + sla_days_suggested` (calendar days or timedelta days—pick one and document).

**E. Rationale**

1. `rationale` is a JSON array of objects each with human-readable `message` and machine `refs` sufficient for UI drill-down to detections, change history, and metrics already exposed by existing GET APIs.

**F. Idempotency**

1. Delete existing `maintenance_recommendations` for `target_inspection_id` before insert **or** upsert keyed by stable natural keys; keep consistent with progression replace policy.

## Phases

Single delivery track: migration + model → rules + engine + worker hook → schemas + GET API + docs + tests. Split into parallel UI/API only if a separate dashboard lands later; not required in this repository today.
