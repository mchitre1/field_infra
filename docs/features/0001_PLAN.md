# Feature 0001: Ingest video/images to analyze

## Context

This is the first vertical slice for IIE: accept **video and image** inspection media so downstream analysis (visual understanding, temporal comparison, prioritization) can run later. Scope matches the product brief’s ingestion layer: FastAPI upload APIs, S3 for raw media, PostgreSQL for inspection metadata, SQS (and eventually Step Functions) to hand work to processing. User-stated requirement, verbatim: **“Ingest video/images to analyze.”**

In-scope for this feature: durable storage of uploaded bytes, persisted inspection/upload records with source and technical metadata, and a queued processing signal so analyzers can consume work. Out-of-scope: defect detection, alignment across time, risk scoring, and dashboard UX beyond what is needed to prove ingestion (optional minimal health/upload-status endpoint only if required for integration tests).

## Algorithms and behavior (step-by-step)

**A. Multipart upload (direct to API)**

1. Receive multipart request: file part + JSON or form fields for metadata (source, optional capture time, optional lat/lon or WKT).
2. Validate: file present; `content_type` in allowlist (e.g. `image/jpeg`, `image/png`, `video/mp4`, `video/quicktime`); aggregate size under configured cap; optional magic-byte sniff if stricter validation is required.
3. Generate `inspection_id` (UUID) before streaming to S3 so the object key is deterministic.
4. Stream file body to S3 with server-side encryption; on failure, return 5xx and do not insert a committed “success” row (or insert `failed` with error detail—pick one policy and test it).
5. Insert PostgreSQL row with S3 location, metadata, `status=stored` (or `queued` if message send is atomic with DB—see below).
6. Build SQS message with `inspection_id`, `s3_uri`, `content_type`, and metadata subset needed by downstream frame extraction.
7. Publish to SQS; if publish fails after S3 put, mark row `failed` or `stored_pending_queue` and expose a retry path (admin or automatic)—document the chosen recovery behavior in code comments only; this plan does not prescribe product SLAs.

**B. Presigned URL flow (alternative for large video)**

1. Client calls `POST /ingest/presign` with metadata and intended `content_type`; server returns `inspection_id`, `upload_url`, fields, and `s3_key`.
2. Client uploads directly to S3.
3. Client calls `POST /ingest/{inspection_id}/complete` (or S3 event → Lambda) to finalize: verify object exists and size, insert/update row, publish SQS message. Same validation as A where applicable.

**C. Metadata extraction (minimal v1)**

1. For images: optionally read dimensions via library (e.g. Pillow) in a background task or in the worker—not blocking the upload response if latency targets require it; persist width/height when available.
2. For video: optionally run `ffprobe` (or equivalent) in the worker stage to persist duration, container format, and frame rate; ingestion API may only store client-supplied capture time until the worker runs.

**D. Idempotency and duplicates**

1. If clients can retry uploads, support optional `Idempotency-Key` header: same key + same tenant returns the same `inspection_id` without duplicating S3 objects—requires a small idempotency store (Redis per brief) or unique constraint table; if Redis is deferred, document “at-least-once uploads may duplicate” and rely on downstream deduplication.

## Phases

Single cohesive feature; split only for parallel work:

- **Phase 1 — Data layer:** Alembic migration(s), SQLAlchemy models, Pydantic schemas, S3 key and bucket configuration, SQS message schema. No HTTP beyond a trivial health check if needed.
- **Phase 2A — API:** FastAPI routes, `storage.py` + `ingest.py` orchestration, validation and error mapping.
- **Phase 2B — Async handoff:** `publisher.py`, optional worker stub or Step Function placeholder subscription, optional presigned URL path if large files are a day-one requirement.

Phase 2A and 2B can proceed in parallel once Phase 1 types and table shape are stable.
