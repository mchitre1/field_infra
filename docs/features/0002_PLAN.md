# Feature 0002: Extract and store frames, timestamps, geospatial/location metadata, and source context

## Context

This feature implements the product-brief requirement, verbatim: **“Extract and store frames, timestamps, geospatial/location metadata, and source context.”**  
Feature 0001 already stores raw media in S3 and inspection rows in PostgreSQL, then publishes SQS jobs. This feature adds the next pipeline step: worker-side frame extraction from image/video uploads, durable frame storage in S3, and normalized frame-level metadata persisted in PostgreSQL for downstream detection and temporal analysis.

In scope: frame extraction, per-frame timestamps, location/source context propagation, and queryable frame metadata records.  
Out of scope: object detection/classification, progression scoring, recommendation UI, and final risk engine logic.

## Algorithms and behavior (step-by-step)

**A. Consume ingest job and lock inspection**

1. Worker receives `IngestJobMessage` (currently published by `publish_ingest_job`).
2. Load inspection by `inspection_id`.
3. Transition inspection to frame-processing state (or set processing marker in metadata) in a single DB transaction.
4. Enforce idempotency:
   - if extraction already completed (`frames_extracted` or existing frame rows), skip/short-circuit based on policy;
   - if partial data exists from prior failure, either resume from highest `frame_index` or clear/rebuild for deterministic output (choose one policy and enforce consistently).

**B. Resolve source media and metadata baseline**

1. Read the source object from `s3://{inspection.s3_bucket}/{inspection.s3_key}`.
2. Build baseline source context from `Inspection`:
   - `source_type`,
   - `site_hint`, `asset_hint`,
   - `capture_timestamp`,
   - `latitude`, `longitude`.
3. For images, extract intrinsic metadata when available (dimensions; EXIF timestamp/GPS if present) and reconcile with inspection-provided values by deterministic precedence rules.
4. For video, inspect stream metadata (duration, fps, dimensions, codec/container) before sampling.

**C. Frame extraction strategy**

1. If source is image MIME (`image/*`):
   - treat as a single frame (`frame_index=0`, `frame_timestamp_ms=0`);
   - store one normalized frame artifact (can be original bytes if already accepted format, or re-encoded canonical format).
2. If source is video MIME (`video/*`):
   - determine effective sampling interval from configured FPS and guards (`max_frames_per_inspection`);
   - compute target frame indices/timestamps from duration and sampling cadence;
   - extract frames sequentially in timestamp order to ensure deterministic indexing.
3. For each extracted frame:
   - compute `frame_timestamp_ms` offset from video start,
   - derive absolute `capture_timestamp` when inspection capture time exists (`inspection.capture_timestamp + offset`),
   - assign location/context fields from baseline (or per-frame values if richer source telemetry exists later),
   - upload frame object to S3 using deterministic key,
   - persist a frame row in DB.

**D. Finalization and failure handling**

1. On success:
   - update inspection summary fields (`frame_count`, optional video metadata) and final extraction status.
2. On recoverable failure (temporary S3/DB/transcode issue):
   - mark inspection extraction status as failed/retryable,
   - persist error detail for observability.
3. On unrecoverable input failure (corrupt media, unsupported encoding):
   - mark inspection as terminal extraction failure with reason.
4. Ensure no silent partial success:
   - either commit frame rows incrementally with resumable policy,
   - or use bounded batches and explicit checkpointing.
