# Feature 0008: Support configurable risk rules by asset type, severity, and environmental conditions

## Context

This feature implements the product-brief functional requirement, verbatim: **“Support configurable risk rules by asset type, severity, and environmental conditions.”**  
Feature 0007 already emits **maintenance recommendations** using **global numeric settings** in `Settings` and a single scoring path in `recommendation_rules.score_zone` / `recommendation_engine.run_recommendations_for_inspection`. There is **no** per-tenant or per-context rule set in PostgreSQL, no structured matching on **asset type** (beyond implicit defect/hazard pooling), **severity** (beyond a single confidence floor), or **environment** (beyond what is accidentally encoded in site strings).

The product brief also states that the decision layer uses **configurable thresholds stored in RDS**; this feature introduces persisted rules as the source of truth, with **settings retained as defaults** when no DB rule applies (backward compatible).

## Files and modules to add or change

**Configuration**

- `backend/app/core/config.py`
  - Optional: `risk_rules_default_org_behavior` (`global_only` \| `merge_global_then_org`), `risk_rules_max_rows_per_eval` (cap rules loaded per recommendation run), path to optional **bootstrap JSON** seed on empty table (dev only).

**Domain and persistence**

- `backend/app/models/risk_rule.py` (new)
  - Persisted declarative rules, for example:
    - `id` (UUID),
    - `org_id` (nullable UUID; `NULL` = applies to all orgs unless superseded by org-specific row with same logical key),
    - `priority` (int, ascending evaluation order: lower = evaluated first, or document opposite and stick to one convention),
    - `enabled` (bool, default true),
    - `name` (short human label for audit),
    - **`match`** (JSON): structured predicates, all of which must hold for the rule to fire (AND semantics within one row), including at minimum:
      - **Asset type:** match on `Inspection.source_type` (`drone` \| `mobile` \| `fixed_camera`) and/or **dominant asset class** in the zone (from `Detection` rows with `detection_type=asset` and `class_name` in a list or prefix match), and/or `Inspection.asset_hint` regex—exact JSON schema versioned (e.g. `match_version: 1`).
      - **Severity:** numeric bands on **model confidence** and/or optional `extra_attributes["severity"]` (string enum or numeric) when present; support `min_confidence`, `max_confidence`, `severity_in` list.
      - **Environmental conditions:** match on `Inspection.latitude`/`longitude` ranges (bounding box), optional `extra_metadata` keys (e.g. `weather_code`, `wind_mps`) when ingestion starts populating them, and/or **frame-level** hints already on `Frame` (`latitude`, `longitude`, `site_hint`) aggregated per zone (max/min/any)—document which v1 uses.
    - **`effect`** (JSON): adjustments applied when the rule matches, e.g. `score_multiplier` (float), `score_add` (float), `confidence_floor_override` (float \| null), `sla_days_override` by priority label map—keep v1 small and extensible.
  - Indexes: (`org_id`, `enabled`, `priority`), partial index on `enabled=true` if supported.
- `backend/app/models/__init__.py`
  - Export `RiskRule`.
- `backend/alembic/versions/0007_*.py` (new; next after `0006` maintenance migration)
  - Create `risk_rules` with FK optional to org if modeled, JSON columns, constraints as needed.

**Rule evaluation**

- `backend/app/services/risk_rule_context.py` (new)
  - Build an immutable **context** object per (`target_inspection`, `asset_zone_id`, aggregated detection stats, change-event counts, progression snapshot) used for JSON predicate evaluation—no DB access inside pure match helpers.
- `backend/app/services/risk_rule_eval.py` (new)
  - Load enabled rules for `org_id` of inspection + global rules per merge policy; iterate in `priority` order; for each rule test `match` against context; accumulate `effect` (document composition: **multiplicative stack** vs **first-wins**—pick one deterministic policy, e.g. multiply all matching `score_multiplier` and sum all `score_add`).

- `backend/app/services/recommendation_engine.py`

## Algorithms and behavior (step-by-step)

**A. Build evaluation context (per zone, per target inspection)**

1. Load target `Inspection` (`org_id`, `source_type`, `asset_hint`, `site_hint`, `latitude`, `longitude`, `extra_metadata`).
2. Partition target `Detection` rows by resolved `asset_zone_id` (same helper as recommendations).
3. Within the zone, compute: list of asset `class_name`s present, max defect/hazard confidence, optional max/min `extra_attributes.severity`, frame geo envelope if any detection-linked `Frame` has coordinates.
4. Serialize into `RiskRuleContext` dataclass or dict with stable keys for JSON matchers.

**B. Load candidate rules**

1. `SELECT … FROM risk_rules WHERE enabled AND (org_id IS NULL OR org_id = :inspection.org_id) ORDER BY priority ASC, id ASC` (or documented order).
2. Apply `risk_rules_default_org_behavior`: e.g. org-specific rules **prepend** or **replace** globals for overlapping `name`—document one policy to avoid double counting.

**C. Evaluate and compose effects**

1. For each rule in order, if `match` JSON evaluates true against context, record `rule_id` and parse `effect`.
2. Combine effects into modifiers `(m_add, m_mul, floor_override, sla_overrides)` using the chosen composition policy.
3. `final_score = (base_score + m_add) * m_mul` with sane clamps (min 0, max configurable) to avoid runaway scores.

**D. Rationale**

1. For each fired rule, append a rationale object `{ "kind": "risk_rule", "message": "…", "refs": { "risk_rule_id": "…", "name": "…" } }` in addition to existing detection/change/progression factors.

**E. SLA interaction**

1. If `effect` includes per-label `sla_days` overrides, merge into the existing `sla_days_for_label` resolution in `recommendation_rules` (post-label) **or** apply a single `sla_multiplier` on computed `sla_days_suggested`—pick one v1 behavior and document.
