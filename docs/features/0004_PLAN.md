# Feature 0004: Align inspections by asset/zone over time to compare state changes

## Context

This feature implements the requirement, verbatim: **ŌĆ£Align inspections by asset/zone over time to compare state changes.ŌĆØ**  
The backend currently supports ingestion, frame extraction, and frame-level detections per inspection. This feature introduces temporal alignment so detections from multiple inspections of the same asset/zone can be matched and compared as a coherent time series.

In scope: asset/zone grouping, cross-inspection frame/detection alignment, persisted alignment/change records, and query APIs for aligned history.  
Out of scope: risk prioritization, SLA recommendations, and user-action feedback loops.

## Algorithms and behavior (step-by-step)

**A. Build comparison cohort**

1. Start from a completed inspection with `detections_ready`.
2. Collect candidate prior inspections in same site/asset cohort using:
   - `org_id`,
   - `site_hint`,
   - `asset_hint`,
   - optional geo proximity window.
3. Order candidates by capture/created timestamp (oldest->newest or rolling baseline policy).

**B. Derive asset/zone keys**

1. For each detection, compute `asset_zone_id` using deterministic precedence:
   - explicit `asset_hint` + class if available,
   - nearest geospatial cluster in site,
   - fallback synthetic key (site + class + coarse grid bucket).
2. Partition baseline/target detections by `asset_zone_id`.

**C. Match detections across time**

1. For each `asset_zone_id`, compare baseline vs target detections with same `detection_type` and compatible class.
2. Compute candidate match score using weighted factors:
   - bounding-box overlap (IoU),
   - centroid distance (normalized),
   - confidence consistency.
3. Apply gating:
   - reject candidates below IoU/confidence thresholds,
   - reject pairs outside geospatial/time tolerance.
4. Resolve one-to-one matches (greedy by score or Hungarian assignment if implemented).
5. Label unmatched rows:
   - baseline unmatched -> `disappeared`,
   - target unmatched -> `appeared`,
   - matched -> `persisted` or other change subtype.

**D. Persist temporal outputs**

1. Upsert/replace `alignment` rows for the inspection pair to keep reruns idempotent.
2. Generate change events from aligned pairs and persist in `change_event`.
3. Update inspection alignment summary fields and set status `alignment_ready`.
4. On failure, set `alignment_failed` and capture error detail in inspection metadata.

**E. API read behavior**

1. Inspection-level alignment endpoint returns aligned pairs with pagination and filters (`asset_zone_id`, `change_type`, class/type).
2. Changes endpoint returns rolled-up events for the inspection.
3. Pairwise compare endpoint returns baseline/target summary plus aligned pair stats.
