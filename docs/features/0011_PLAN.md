# Feature 0011: Maintain traceable inspection history and decision logs per asset/zone

## Context

This feature implements the product-brief functional requirement, verbatim: **“Maintain traceable inspection history and decision logs per asset/zone.”**  
Today the backend persists **inspections** (current `status`, counts, metadata) in `backend/app/models/inspection.py`, **per-inspection** pipeline artifacts (frames, detections, alignment, change events, progression metrics, **replace-semantics** `maintenance_recommendations` in `backend/app/services/recommendation_engine.py`), **zone-scoped temporal** rows surfaced by `backend/app/services/anomaly_timeline.py` (`build_timeline` → `GET /ingest/timeline` in `backend/app/api/routes/ingest.py`), **operator issue workflow** via `IssueState` / `IssueStateEvent` in `backend/app/models/issue_state.py` and `upsert_issue_state` in `backend/app/services/issue_state_service.py`, and **operator outcomes** via `OutcomeFeedback` in `backend/app/models/outcome_feedback.py` and `submit_outcome_feedback` in `backend/app/services/outcome_feedback_service.py`. There is **no** unified, append-only **decision log** keyed for audit export per `asset_zone_id`, and **no** first-class **inspection lifecycle history** (status transitions are only inferable from the current row plus scattered `extra_metadata` errors—not an ordered, queryable log).

This plan adds **durable, append-only log tables** plus **write hooks** at the points where decisions or lifecycle changes occur, and a **read API** that lists them for a zone (and optionally scopes by `org_id` / time / `inspection_id`). It does **not** replace **`maintenance_recommendations`** replace semantics with versioning (that would be a larger schema change); instead, each recommendation pass **materializes log entries** with denormalized score/label/rationale snippets so history survives the next `delete` in `run_recommendations_for_inspection`.

## Files and modules to add or change

**Domain and persistence (new)**

- `backend/app/models/zone_decision_log.py` (new; name fixed in implementation)
  - Append-only rows: `id`, `created_at` (server default), `org_id` (nullable UUID, indexed), `asset_zone_id` (string, indexed), optional `issue_key` (nullable string), optional `inspection_id` (FK `inspections.id`, `ondelete=SET NULL`), **`event_type`** constrained string (v1 set such as `issue_state_transition`, `operator_outcome`, `maintenance_recommendation`, `inspection_status_change`, optional `note` / `system` for future worker milestones).
  - **`payload`**: JSON blob with human-readable `summary`, stable **`refs`** (UUID strings for `issue_state_event_id`, `outcome_feedback_id`, `maintenance_recommendation_id` when still present, `inspection_id`, `issue_key`, prior recommendation snapshot fields mirroring `OutcomeFeedback` capture pattern so rows stay interpretable after FK targets disappear).
  - Optional nullable FKs (`issue_state_event_id`, `outcome_feedback_id`, `maintenance_recommendation_id`) all `ondelete=SET NULL` so logs outlive deletes; **payload remains authoritative** when FKs null out.
  - Composite / secondary indexes aligned to export: `(asset_zone_id, created_at desc)`, `(org_id, asset_zone_id, created_at)`, `(inspection_id, created_at)`.

- `backend/app/models/inspection_history_event.py` (new) **or** fold into `zone_decision_log` with `event_type=inspection_status_change` only—pick one in implementation to avoid duplicate streams. If separate: columns `inspection_id` (FK), `from_status`, `to_status`, `source` (`worker` \| `api` \| `system`), `context` JSON, `created_at`; indexes `(inspection_id, created_at)`, `(created_at)`.

- `backend/alembic/versions/` (new revision after current head)
  - Create the new table(s), FKs, indexes; import new models in `backend/alembic/env.py` following the `issue_state` / `outcome_feedback` pattern.

- `backend/app/models/__init__.py`
  - Export new model(s).

- `backend/tests/conftest.py`
  - Register new models for SQLite metadata in tests.

**Services**

- `backend/app/services/zone_decision_log_service.py` (new)
  - `append_zone_decision_log(db, *, org_id, asset_zone_id, event_type, payload, inspection_id=None, issue_key=None, issue_state_event_id=None, outcome_feedback_id=None, maintenance_recommendation_id=None) -> row` — single insert, no updates.
  - `list_zone_decision_log(db, *, org_id, asset_zone_id, inspection_id=None, event_type=None, created_from, created_to, limit, offset) -> (rows, total)` — same org-scoping convention as `list_issue_states` / `list_outcome_feedback` (`org_id` filter maps to `org_scope` on issue rows elsewhere; here use nullable `org_id` column on log rows **as captured at write time** for simple filtering, document mismatch vs `IssueState.org_scope` string for global org).

**Write hooks (call `append_zone_decision_log` after the originating transaction commits or in the same transaction before commit—pick one and stay deadlock-safe)**

- `backend/app/services/issue_state_service.py` — inside `upsert_issue_state`, after inserting `IssueStateEvent` (both first insert and transition paths), append a log row with `event_type=issue_state_transition`, `refs.issue_state_event_id`, `from_state`/`to_state`, `actor`, `issue_key`, `asset_zone_id`.

- `backend/app/services/outcome_feedback_service.py` — after successful insert in `submit_outcome_feedback`, append `event_type=operator_outcome` with `refs.outcome_feedback_id`, `outcome_kind`, `outcome_code`, optional inspection/detection refs from payload.

- `backend/app/services/recommendation_engine.py` — in `run_recommendations_for_inspection`, after each `MaintenanceRecommendation` row is flushed/added (inside the same try that commits the batch), append one log row per zone line written with `event_type=maintenance_recommendation`, denormalized `priority_label`, `priority_score`, truncated rationale head, `refs.maintenance_recommendation_id` and `target_inspection_id`. **Do not** write logs for zones skipped with score `0` and no factors unless the engine already emits a row for them—match whatever rows are actually persisted today.

- **Inspection lifecycle (if separate `inspection_history_event` table):** add small helper `append_inspection_history_event` and call from worker stages when `Inspection.status` is assigned in `backend/app/services/detection_pipeline.py`, `backend/app/services/alignment.py`, `backend/app/services/progression.py`, `backend/app/services/recommendation_engine.py` (success/failure metadata paths), `backend/app/services/ingest.py` / presign complete flows in `backend/app/api/routes/ingest.py`—only when the status value **changes** to avoid log spam; include `inspection_id`, `from_status`, `to_status`, optional error key from `extra_metadata`. Alternatively emit **`inspection_status_change`** rows into the same `zone_decision_log` with `asset_zone_id` null and `inspection_id` set, and filter the read API—choose one representation and document.

## Algorithms and behavior (step-by-step)

**A. Append on issue state change**

1. `upsert_issue_state` completes `IssueStateEvent` insert (existing behavior).
2. Build `payload`: `{ "summary": "...", "refs": { "issue_state_id", "issue_state_event_id", "from_state", "to_state", "actor" } }`.
3. `append_zone_decision_log` with `org_id`, `asset_zone_id`, `issue_key`, optional `last_target_inspection_id` mirrored as `inspection_id` only when product wants cross-link—**optional**; if omitted, keep `inspection_id` null for pure issue events.

**B. Append on outcome submit**

1. After `OutcomeFeedback` insert succeeds, append with `refs.outcome_feedback_id` and copy of key outcome fields into `payload`.

**C. Append on maintenance recommendation write**

1. For each `MaintenanceRecommendation` inserted in the final loop of `run_recommendations_for_inspection`, append one log row keyed by `asset_zone_id` and `inspection.id` as `inspection_id`, `org_id` from inspection, `payload` containing `priority_rank`, `priority_label`, `priority_score`, `action_summary`, and a **bounded** rationale excerpt (character cap) to control row size.

**D. List for audit**

1. Validate `asset_zone_id` non-empty; optional `org_id` narrows to rows whose stored `org_id` matches (including NULL semantics for global—document whether global logs use `org_id IS NULL` only).
2. `ORDER BY created_at DESC, id DESC` with `limit`/`offset`; return `total` via subquery count pattern used in `list_issue_states`.

**E. Inspection history (if separate events table)**

1. On each status mutation, compare new status to previous; if unchanged, skip.
2. Insert `inspection_history_event` row with monotonic `created_at`.
3. `GET` inspection history by `inspection_id` ascending for a linear inspection trace; optionally also project key milestones into `zone_decision_log` for cross-zone dashboards (defer duplicate projection if it complicates v1).

## Phases

**Phase 1 — Data layer + append helpers + hooks**  
Models, Alembic, `zone_decision_log_service.append_*`, hooks from `issue_state_service`, `outcome_feedback_service`, `recommendation_engine`, tests proving rows appear.

**Phase 2 — Read API + documentation**  
`GET` route, Pydantic schemas, `INGEST_API.md`, and optional inspection-only history route if Phase 1 introduced a separate `inspection_history_event` table.

Phases 2A/2B parallel UI are out of scope (backend-only repo).

## Explicitly out of scope

- Rewriting **`maintenance_recommendations`** to soft-delete or versioned rows (would reduce need for denormalized payloads but is a different feature).
- Merging **`zone_decision_log`** into **`GET /ingest/timeline`** response shape in v1 (timeline stays change/progression unless explicitly extended later).
- Cryptographic signing, WORM storage, or legal hold retention policies (operational DB only).
