# Feature 0009: Allow users to update issue state (fixed, monitoring, deferred, ignored)

## Context

This feature implements the product-brief functional requirement, verbatim: **“Allow users to update issue state (fixed, monitoring, deferred, ignored).”**  
The backend persists **detections**, **alignment** / **change events**, **progression metrics**, **maintenance recommendations**, and **risk rules**, but there is **no** first-class **issue** record or workflow state that operators can mutate independently of pipeline-generated rows. User intent (work completed, watch-only, postponed, dismissed) must be stored, queryable, and auditable without overwriting raw inspection artifacts.

This plan does **not** implement the adjacent brief items **“Capture user outcomes and feed them back into model/risk-score refinement”** or full **decision logs** beyond what an append-only state history provides; those remain follow-on features.

## Files and modules to add or change

**Domain and persistence**

- `backend/app/models/issue_state.py` (new)
  - Canonical **issue** identity for v1 (stable across inspections): e.g. `org_id` (nullable UUID), `asset_zone_id` (string, indexed), `issue_key` (string, indexed) — deterministic key derived from `detection_type` + normalized `class_name` (and optionally `issue_subtype` string default `default`) so the same logical defect class in a zone maps to one row.
  - `state` column: enum or constrained string **`fixed` \| `monitoring` \| `deferred` \| `ignored`** (store lowercase; validate on write).
  - `notes` (optional `Text`), `updated_by` (optional `String(255)` for operator id/email placeholder until auth exists).
  - `last_target_inspection_id` (optional FK `inspections.id`) — last inspection that surfaced this issue (updated when recommendations run or when explicitly linked).
  - `created_at`, `updated_at` (`DateTime(timezone=True)`).
  - Unique constraint on `(org_id, asset_zone_id, issue_key)` with NULL-safe policy for `org_id` (document SQLite vs PostgreSQL behavior; use sentinel `org-unknown` string if NULL-uniqueness is problematic).
- `backend/app/models/issue_state_event.py` (new, optional but recommended for brief traceability)
  - Append-only: `id`, `issue_state_id` (FK), `from_state`, `to_state`, `actor` (optional), `created_at`, optional `context` JSON (e.g. `inspection_id`, `recommendation_id`).
- `backend/app/models/__init__.py`
  - Export new models.
- `backend/alembic/versions/0008_*.py` (new; next after `0007` risk_rules migration)
  - Tables, indexes on (`asset_zone_id`, `state`), (`org_id`, `asset_zone_id`).

**Schemas**

- `backend/app/schemas/issues.py` (new)
  - `IssueState` literal type / enum for API.
  - `IssueUpsertRequest` (`state`, optional `notes`, optional `updated_by`, optional `last_target_inspection_id`).
  - `IssuePublic`, `IssueListResponse` (paginated), `IssueStateEventPublic` if history is exposed.

**Services**

- `backend/app/services/issue_key.py` (new)
  - `build_issue_key(detection_type: str, class_name: str, *, subtype: str = "default") -> str` — normalized, stable, documented format (e.g. `defect:crack:default`).
- `backend/app/services/issue_state_service.py` (new)
  - `upsert_issue_state(db, org_id, asset_zone_id, issue_key, body) -> IssueStateRow` — insert or update `state`/`notes`/`updated_by`, touch `updated_at`, optionally append `IssueStateEvent`.
  - `list_issues(db, filters...)` for GET.

**API**

- `backend/app/api/routes/issues.py` (new) + `main.py` include **or** prefixes under `/ingest/issues`:
  - `PUT` or `PATCH /issues/{org_segment}/{asset_zone_id}/{issue_key}` — URL-encoded path segments for `asset_zone_id` / `issue_key` (use query params instead if path encoding is too fragile: `PUT /issues/state?org_id=&asset_zone_id=&issue_key=` — **pick one** approach and document; query form is simpler for clients).
  - Recommended v1: **`PUT /issues/state`** with JSON body `{ "org_id", "asset_zone_id", "issue_key", "state", "notes?", "updated_by?", "last_target_inspection_id?" }` for upsert.
  - `GET /issues` — filter by `org_id`, `asset_zone_id`, `state`, pagination.

**Optional linkage**

- `backend/app/models/maintenance_recommendation.py`
  - Optional nullable `issue_state_id` FK (defer if scope tight): lets a recommendation row point at the user-owned issue shell; otherwise recommendations and issues remain loosely coupled by `asset_zone_id` + key derived from rationale.

**Documentation**

- `docs/INGEST_API.md` (or new `docs/ISSUES_API.md` if preferred)
  - Document `issue_key` derivation, allowed `state` values, upsert semantics, and event history if exposed.

**Tests**

## Algorithms and behavior (step-by-step)

**A. Issue identity**

1. Client supplies `asset_zone_id` (same string family as alignment/recommendations) and `issue_key` **or** supplies `detection_type` + `class_name` + optional `subtype` from which the server computes `issue_key` via `issue_key.py` (document one supported path for v1; recommend server-side key build to avoid client skew).

**B. Upsert**

1. Resolve `org_id` (nullable → treat as global bucket per unique constraint policy).
2. `SELECT` existing row by unique key; if absent `INSERT` with defaults; if present `UPDATE` `state`, optional fields, `updated_at`.
3. If `IssueStateEvent` is in scope: insert row with `from_state` / `to_state` (use `NULL` or `"open"` for first insert—document).

**C. Validation**

1. Reject unknown `state` strings with **422**.
2. Optionally validate `last_target_inspection_id` exists when provided.

**D. Read**

1. `GET /issues` returns latest row per issue; optional `include_events=true` adds ordered history.

## Phases

Single track: migration + models → issue_key + service → HTTP routes + docs + tests. `IssueStateEvent` can ship in the same migration or immediately after if timeboxed.
