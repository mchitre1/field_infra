# Feature 0003: Detect and classify assets, defects, and environmental hazards in each inspection

## Context

This feature implements the product-brief requirement, verbatim: **“Detect and classify assets, defects, and environmental hazards in each inspection.”**  
The repository already has ingestion and frame extraction. This feature adds model inference on extracted frames, persists structured detections, and exposes queryable results for each inspection/frame.

In scope: frame-level detection/classification, defect/hazard labeling, confidence + geometry persistence, and read APIs for inspection results.  
Out of scope: temporal comparison/progression, risk scoring/prioritization, and recommendation generation.

## Algorithms and behavior (step-by-step)

**A. Detection pipeline trigger**

1. Worker receives inspection job after frames are extracted (same worker flow or chained phase).
2. Mark inspection detection state as `processing_detections` (or equivalent metadata flag).
3. Load all frames for the inspection ordered by `frame_index`.

**B. Per-frame inference**

1. Fetch frame bytes from frame S3 object.
2. Run model inference (YOLO; optional SAM refinement path for segmentation geometry).
3. Normalize raw model outputs:
   - bounding boxes to canonical coordinate format,
   - confidence values to float range [0,1],
   - class IDs/labels to taxonomy labels.
4. Apply post-processing:
   - confidence filtering (`>= inference_confidence_threshold`),
   - class-specific overrides (if configured),
   - NMS to remove duplicate overlaps.

**C. Classification into required groups**

1. Map each normalized label into one of:
   - `asset`,
   - `defect`,
   - `environmental_hazard`.
2. For unknown/unmapped labels:
   - either drop and log,
   - or persist as `asset`/`defect` fallback class per explicit policy (choose one and keep deterministic).

**D. Persistence and idempotency**

1. Before writing new detections, enforce idempotent policy:
   - clear existing detection rows for the inspection and rewrite, or
   - resume from frame checkpoints.
2. Insert detection rows frame-by-frame (batched inserts allowed).
3. Update inspection summary fields:
   - `detection_count`,
   - detection completion status (`detections_ready` on success).
4. On failure:
   - mark `detections_failed`,
   - persist error details for debugging/retry.

**E. API query behavior**

1. Inspection-level endpoint returns detections with pagination.
2. Optional filters:
   - `detection_type`,
   - `class_name`,
   - `min_confidence`,
   - `frame_id`.
3. Frame-level endpoint returns detections only for a single frame.
