# Feature 0006: Generate change maps, anomaly timelines, and trend summaries

## Context

This feature implements the product-brief functional requirement, verbatim: **“Generate change maps, anomaly timelines, and trend summaries.”**  
The backend already persists **frames** (S3 keys, optional `width`/`height`), **detections** (normalized bbox, class, type, `frame_id`), **alignment pairs** (`change_type` in `appeared` \| `disappeared` \| `persisted`, `alignment_score`, `asset_zone_id`), **change events** (rolled-up `event_type` + `event_payload` per target inspection), and **progression metrics** (scalar `metric_name` / `value` / `payload` per baseline→target step). There are read APIs for alignment pairs, change-event lists, pairwise compare, and per-inspection progression list/summary, but **no** dedicated artifacts that (a) spatially encode change for visualization, (b) unify temporal “anomaly” signals across sources for one view, or (c) summarize metric **trends across multiple inspections** for an `asset_zone_id` (or site cohort).

## Files and modules to add or change

**Schemas (new module)**

- `backend/app/schemas/temporal_insights.py` (new), or three small files under `backend/app/schemas/` if preferred for imports:
  - **Change map:** response model for a feature collection in **normalized image coordinates** (0–1), e.g. `ChangeMapFeature` (geometry type, bbox corners, properties: `change_type`, `class_name`, `detection_type`, `alignment_pair_id`, `baseline_detection_id`, `target_detection_id`, `frame_id` per side when resolvable), `ChangeMapLayer` grouped by `frame_id` or flat list with `frame_id` on each feature, envelope `ChangeMapResponse` including `baseline_inspection_id`, `target_inspection_id`, optional `asset_zone_id` filter echo, optional presigned frame URLs (if wired through existing storage helper—see services).
  - **Anomaly timeline:** `TimelineEntry` with stable `entry_kind` discriminator (e.g. `change_event`, `progression_metric`, optional `first_detection` if implemented), `effective_at` (datetime), `inspection_id`, `asset_zone_id`, `severity` or `priority` optional scalar, `summary` string, `refs` (UUIDs / names for drill-down).
  - **Trend summary:** `TrendSeriesPoint` (`effective_at`, `inspection_id`, `value`, optional `metric_unit`), `TrendSummaryResponse` with `asset_zone_id`, `metric_name`, `points` ordered ascending by time, aggregates (`min`, `max`, `mean`, `latest`, `delta_first_to_latest`, optional `simple_slope_per_day` if enough points and policy allows).

**Services**

- `backend/app/services/change_map.py` (new)
  - Pure assembly from DB rows: load `Alignment` rows for a given (`baseline_inspection_id`, `target_inspection_id`) with optional `asset_zone_id`; join `Detection` for baseline/target sides; join `Frame` for `frame_id`(s) and dimensions; emit normalized bbox features and optional pairwise “delta” metadata (e.g. IoU from `alignment_score`, disappeared/appeared flags—**no** new CV; reuse stored `alignment_score`).
- `backend/app/services/anomaly_timeline.py` (new)
  - Build ordered timeline for a scope: required **cohort filter** (e.g. `asset_zone_id` + optional `org_id` / `site_hint` / date range) documented in OpenAPI.
  - Query `ChangeEvent` joined to **target** `Inspection` for `capture_timestamp` / `created_at`.
  - Query `ProgressionMetric` joined to **target** `Inspection` for the same effective time; map each row to a timeline entry (metric name + value in summary/refs).
  - Sort by `effective_at`, then stable tie-breaker (`inspection_id`, `id`).
  - Optional v1 extension: “first high-confidence detection” entries per (`asset_zone_id`, `class_name`) using `Detection` + inspection time—only if explicitly needed for parity with “anomaly”; otherwise omit to keep scope tight.
- `backend/app/services/trend_summary.py` (new)
  - For fixed `asset_zone_id` and `metric_name`, select all `ProgressionMetric` rows (optionally constrained by `org_id` via join on `Inspection`), order by target inspection effective time.
  - Compute aggregates and simple derived stats in Python (or single SQL window for min/max—either is fine); document time basis (same as progression: `capture_timestamp` else `created_at` on **target** inspection).

**API**

- `backend/app/api/routes/ingest.py` **or** `backend/app/api/routes/temporal_insights.py` (new) + register router in `backend/app/main.py`:
  - `GET /ingest/compare/change-map` — query params: `baseline_inspection_id`, `target_inspection_id`, optional `asset_zone_id`, optional `frame_id` to restrict one frame; returns change-map schema.
  - `GET /ingest/timeline` — query params: `asset_zone_id` (required unless alternate cohort key is defined), optional `org_id`, `site_hint`, `from`/`to` on effective time, optional `event_type` / `metric_name` filters; returns list of `TimelineEntry`.
  - `GET /ingest/trends` — query params: `asset_zone_id`, `metric_name`, optional `org_id`, optional time window; returns `TrendSummaryResponse`.
- `docs/INGEST_API.md`
  - Document the three endpoints, coordinate system for change maps (normalized 0–1), and definition of `effective_at` for timeline/trends.

**Tests**

- `backend/tests/test_change_map_service.py` (new): fixture alignments + detections + frames → expected feature count and `change_type` coverage.
- `backend/tests/test_anomaly_timeline_service.py` (new): mixed change events + progression rows → sorted order and field presence.
- `backend/tests/test_trend_summary_service.py` (new): known metric series → aggregates and slope/delta behavior edge cases (single point, two points, N points).
- `backend/tests/test_ingest_api.py` (extend): HTTP tests for the new routes (404 on missing inspections, auth/org scoping if applicable).

**Configuration (only if needed)**

- `backend/app/core/config.py`

## Algorithms and behavior (step-by-step)

**A. Change map**

1. Validate both inspections exist; optionally require target in `alignment_ready` or allow read for any pair that already has `alignment_pairs` rows.
2. Select `Alignment` rows for the baseline/target pair; apply `asset_zone_id` and/or `frame_id` filters.
3. For each row, resolve baseline and target `Detection` by FK; read `bbox_*` (already normalized in pipeline—keep consistent with detection writer).
4. Emit one or two rectangular features per pair (baseline rect, target rect) with shared `alignment_pair_id` and styling properties, **or** a single “pair” feature with `geometry` listing both boxes in `properties`—pick one representation and document it for the React dashboard consumer.
5. Join `Frame` on each detection’s `frame_id` to attach `width`/`height` and S3 location if the API includes optional presigned URLs for client overlay (reuse `app.services.storage` patterns used elsewhere for frames).

**B. Anomaly timeline**

1. Parse cohort filters; reject requests with insufficient scope (e.g. missing `asset_zone_id` if that is the v1 key).
2. **Change events:** `select ChangeEvent join Inspection on inspection_id = Inspection.id` where `ChangeEvent.asset_zone_id` matches (and org/site filters propagate via `Inspection.org_id` / `site_hint` as available).
3. **Progression metrics:** `select ProgressionMetric join Inspection on target_inspection_id = Inspection.id` with same cohort filters on `ProgressionMetric.asset_zone_id` and inspection fields.
4. Map each row to `TimelineEntry` with `effective_at = coalesce(Inspection.capture_timestamp, Inspection.created_at)` (match precedence used in progression docs).
5. Stable-sort entries; apply `limit` from settings/query with documented behavior (truncate tail vs error).

**C. Trend summary**

1. Load all matching `ProgressionMetric` rows for (`asset_zone_id`, `metric_name`); join target `Inspection` for effective timestamps.
2. Build `points` sorted by `effective_at`.
3. Aggregates: `min`/`max`/`mean` over values; `latest` = last point’s value; `delta_first_to_latest` = last − first when ≥ 2 points.
4. Optional `simple_slope_per_day`: least-squares or `(last − first) / span_days` when `span_days` ≥ configured minimum (reuse spirit of `progression_min_time_delta_seconds` or a new `trend_min_span_days`—reference config only, no code in this plan).

## Phases

Single cohesive slice: new schemas + three services + three GET endpoints + docs + tests. Optional follow-up (not required for this plan): persist pre-rendered raster change maps to S3 and a small `change_map_artifacts` table for heavy sites—only if profiling shows on-the-fly assembly is too slow.
